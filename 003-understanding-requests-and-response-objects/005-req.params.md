## What is req.params

### 1. Why Do We Need URL Parameters?

So far, weâ€™ve worked with **fixed URLs** like:

- `/`
- `/users`

But real applications need **dynamic data**, such as:

- User with ID `5`
- Product with ID `123`
- Order with ID `abc123`

Creating separate routes like:

- `/users/1`
- `/users/2`
- `/users/3`

is impossible and wrong.

ðŸ‘‰ **This is why URL parameters exist.**

They allow **one route** to handle **many dynamic values**.

### 2. Fixed vs Dynamic Parts of a URL

Example URL:

    /users/42

Breakdown:

- `/users` â†’ fixed part
- `42` â†’ variable part

That variable part becomes available in Express as:

- `req.params`

---

### 3. What Are URL Parameters?

**URL parameters** are:

- Named **placeholders**
- Defined inside the **URL path**

Example route:

    /users/:id

Here:

- `:id` â†’ URL parameter
- `id` â†’ parameter name
- Value comes from the actual URL

---

### 4. First Complete Example

Add this route:

    app.get("/users/:id", (req, res) => {
      console.log(req.params);
      res.send("Check terminal");
    });

Now open your browser:

    http://localhost:3000/users/42

Terminal output:

    { id: '42' }

---

### 5. Break This Down VERY Slowly

#### `/users/:id`

- `/users` â†’ fixed path
- `:id` â†’ dynamic placeholder

Express logic:

> â€œWhatever comes here, store it under the name `id`â€

---

#### `req.params`

- `req.params` is an **object**
- Keys â†’ parameter names
- Values â†’ actual values from the URL

Example:

    /users/42

Results in:

    req.params = {
      id: "42"
    }

---

### 6. Accessing a Single Parameter

Example:

    app.get("/users/:id", (req, res) => {
      const userId = req.params.id;
      res.send(`User ID is ${userId}`);
    });

Browser output:

    User ID is 42

---

### 7. Multiple URL Parameters

You can define **more than one** parameter in a route.

Example:

    app.get("/users/:userId/posts/:postId", (req, res) => {
      console.log(req.params);
      res.send("Check terminal");
    });

URL:

    /users/10/posts/99

Output:

    {
      userId: '10',
      postId: '99'
    }

Order matters and names matter.

---

### 8. Important Beginner Rule 

**URL parameters are ALWAYS strings.**

Even if they look like numbers:

    /users/42

    typeof req.params.id // "string"

If you need a number:

    const id = Number(req.params.id);

Or:

    const id = parseInt(req.params.id, 10);

---

### 9. Common Confusion: Params vs Query Strings

âŒ This is **NOT** a URL parameter:

    /users?id=42

That is a **query string**, accessed via:

- `req.query`

URL parameters live in the **path**, not after `?`.

---

### 10. When Should You Use `req.params`? 

Use `req.params` when:

- The resource is identified by the URL itself
- You are accessing **one specific thing**

Common patterns:

- `/users/5`
- `/products/99`
- `/orders/abc123`

Rule of thumb:

> **If it identifies a resource â†’ use URL params**

---

### 11. How Express Matches Routes

Express matches routes **top to bottom**.

This order matters:

    app.get("/users/:id", ...);
    app.get("/users/profile", ...);

Request:

    /users/profile

Express may treat this as:

    id = "profile"

To avoid this:

- Place **specific routes first**
- Place **dynamic routes later**

Correct order:

    app.get("/users/profile", ...);
    app.get("/users/:id", ...);

---

### 12. Advanced: Validation of URL Parameters

Never trust URL parameters blindly.

Example:

    app.get("/users/:id", (req, res) => {
      const id = Number(req.params.id);

      if (Number.isNaN(id)) {
        return res.status(400).send("Invalid user ID");
      }

      res.send(`User ID is ${id}`);
    });

This prevents invalid input like:

    /users/abc

---

### 13. Advanced: Optional Parameters

Express supports optional parameters:

    /users/:id?

This means:

- `/users`
- `/users/42`

Both match the same route.

Use carefully â€” can reduce clarity.

---

### 14. Advanced: Params vs Route Design

Good REST design:

- `/users/:id` â†’ one user
- `/users` â†’ all users

Bad design:

- `/getUserById/:id`

URL parameters help keep APIs **clean and predictable**.
